## 2. 当前 UART TX 实现事实

### 2.1 底层 DMA 入口

- 实现：`BSP/Peripherals/bsp_uart.c` 中的 `bsp_uart_send_dma()`。
- 发送许可由 `s_uart_tx_complete` 控制；DMA 完成和 UART EOT 由 `bsp_uart_irq_handler()` 更新。
- 调用入口先判断 `s_uart_tx_complete`，之后将其置为 `false` 并配置 DMA。
- 该“检查空闲 → 占用发送状态 → 配置 DMA”的组合未被任务互斥或临界区整体保护。因此，多个可抢占任务先调用 `bsp_uart_tx_idle()` 再调用 `bsp_uart_send_dma()` 时存在竞态窗口。
- 内部 bounce buffer 为 `512` 字节：长度不超过 512 字节时会复制到 `s_dma_tx_buf`；长度超过 512 字节时当前实现改为直接将调用者指针交给 DMA。后一分支会使调用者缓冲区生命周期成为 DMA 正确性的前提。
- `s_uart_tx_dma_done` 在 DMA 完成中断中置位，但在当前发送准入逻辑中没有用于决定是否可提交新的发送；新的发送以 UART EOT 为最终可重用时点。

### 2.2 阻塞输出入口

- 实现：`Config/board.c` 中的 `fputc()`。
- 当前实现轮询 `DL_UART_isBusy(UART_0_DEBUG_INST)`，最多循环 80,000 次，然后直接写 TXDATA。
- 该路径未与 DMA 发送服务协调；运行态 `printf` 与 DMA 遥测共用 UART0 时存在输出交织与实时性风险。

## 3. CodeGraph 调用关系审计

`bsp_uart_send_dma()` 当前有 3 个直接调用者：

| 调用者 | 文件 | 执行上下文 | 当前缓冲区 | 当前行为 | 风险 |
|---|---|---|---|---|---|
| `imu_send_dma()` | `Application/Task/task_imu.c` | IMU 任务，优先级 4，约每 200 ms | 静态 `g_dma_buf[512]` | 先 `bsp_uart_tx_idle()`，后格式化并调用 DMA | 与其他任务竞争“检查后发送”；文本 FireWater 帧可能与日志交织 |
| `menu_data_output_loop()` | `Application/Task/task_menu.c` | 菜单任务，优先级 2，菜单数据模式下周期运行 | 栈上 `tx_buf[180]` | 先 `bsp_uart_tx_idle()`，后调用 DMA | 调用层检查与底层状态置位之间可被高优先级任务抢占 |
| `app_debug_encoder_stream()` | `Application/app_debug.c` | 菜单/调试路径 | 栈上 `buf[48]` | 先 `bsp_uart_tx_idle()`，后调用 DMA | 同样存在竞争；并同时调用 `printf` 输出控制文本 |

### 3.1 当前缓冲区生命周期结论

在当前三条直接 DMA 调用路径中，帧长度上限均不超过底层 512 字节 bounce buffer：

- IMU：`g_dma_buf[512]`；
- 菜单：`tx_buf[180]`；
- 编码器调试：`buf[48]`。

因此，**当前这些已审计调用点**在长度不超过 512 字节时，源缓冲区会先复制到 BSP 内部缓冲区，栈上 `tx_buf` / `buf` 不会直接被 DMA 持有。

但该结论不能视为 API 的安全保证：一旦新增帧长度大于 512 字节的调用点，`bsp_uart_send_dma()` 会直接使用调用者指针，异步 DMA 可能读取已失效或被改写的内存。第一阶段 TX Service 必须禁止该隐式降级路径，改为对所有可发送帧建立明确的静态所有权。

## 4. 当前已确认的风险

| 编号 | 风险 | 严重度 | 依据 | 第一阶段处置 |
|---|---|---:|---|---|
| UART-R1 | 多任务“先查 idle、后启动 DMA”的竞争窗口 | P0 | 3 个任务均在调用层检查 idle；底层 check-and-set 不具备整体原子性 | 由唯一 TX 任务独占 DMA |
| UART-R2 | `printf` 轮询与 DMA 共用 UART0 | P0 | `board.c` 的 `fputc()` 不经 DMA 仲裁 | 运行态日志改为异步 TX Service |
| UART-R3 | 文本日志与 VOFA 遥测混线 | P0 | IMU、菜单和调试输出使用同一 UART0 | 先统一发送服务；后续确定协议或物理串口隔离 |
| UART-R4 | 大于 512 字节时 DMA 直接引用调用者内存 | P1 | `bsp_uart_send_dma()` 的 `len > 512` 分支 | 新服务使用固定容量消息块；超长帧显式拒绝或分片 |
| UART-R5 | 现有 UART 统计仅覆盖部分 ISR 调试数据 | P1 | `UART_DEBUG_STATS` 默认关闭，未覆盖队列、丢帧、超时语义 | 先添加只读运行时诊断，再改变发送架构 |

## 5. 未在本次审计中做出的假设

下列事项必须通过实际工程配置和上板测试确认，不能根据静态代码猜测：

- UART0 的真实波特率、有效吞吐、FIFO 深度和 DMA 完成时序；
- 17 通道 IMU 文本帧的实际最长长度；
- 菜单数据模式的真实发送周期和持续时长；
- PC 端停止读取、慢读取或协议异常时的 UART 行为；
- UART DMA 与 `fputc()` 并发时是否已在板上复现帧损坏；
- 现有任务栈、堆与 CPU 时间预算是否还能容纳新增静态 TX 服务。

## 6. 进入 UART TX Service 改造前的准入门槛

P0-1 只读诊断已经实现，但在新增 TX 队列、TX Task 或改变 DMA 发送仲裁前，必须完成以下确认：

1. 在 Keil/Arm Compiler 目标环境成功构建当前基线；
2. 记录启动日志、IMU 遥测、菜单输出与编码器调试输出的实际帧长度；
3. 在电机安全条件下进行至少一次 UART 背压测试：主机暂停或显著降低串口读取能力；
4. 记录控制任务和 IMU 任务运行时长、是否有明显周期抖动或异常；
5. 明确 TX 队列的 RAM 预算、最大帧容量、队列深度与满队列丢弃策略；
6. 明确只有 TX Task 可调用 `bsp_uart_send_dma()` 的接口收口方案；
7. 确认新增诊断不会在控制任务、IMU 任务或 ISR 中调用 `printf`。

若其中任何一项无法确认，则停止进入队列化改造，先补充测量能力或硬件测试条件。

## 8. P0-1 只读诊断实现记录

已在 `BSP/Peripherals/bsp_uart.h/.c` 新增以下只读诊断能力：

- `bsp_uart_tx_diag_t`：记录 DMA 启动、DMA 完成、EOT 完成、busy 拒绝、空指针拒绝、零长度提交、超 512 B 提交、最近长度和最大长度；
- `bsp_uart_get_tx_diag()`：仅允许任务上下文调用，返回不重置状态的诊断快照；
- 诊断代码不改变 `bsp_uart_send_dma()` 的发送准入条件、DMA 配置顺序、UART 协议、任务优先级或控制参数；
- 诊断计数仅用于观测，不得作为控制或安全决策输入；因现有多生产者竞态尚未消除，计数为 best-effort，不能被误解为严格的并发证明。

已执行目标为 Cortex-M0+ 的 Clang 语法检查。检查使用了仅位于工作区外部临时目录的 `reent.h`、`math.h`、`string.h` 兼容桩，以弥补本机缺失的 Arm/Keil C 库头文件；该检查验证了本次修改的 C 语法和头文件集成，但**不替代 Keil/Arm Compiler 的正式构建和上板测试**。
