## 1. 目的与严格边界

本协议只验证临时命令 `encdiag` 输出的**只读编码器快照**与实际物理轮之间的映射关系，并归档后续整改前必须保留的原始证据。

本工作包明确不做以下事项：

- 不改变 PID、前馈、位置控制、滤波、PWM、DRV8870、电机使能逻辑或任何控制参数；
- 不改动 `Config/empty.syscfg`、生成的 TI 配置文件、编码器方向宏或 GPIO 映射；
- 不发送 VOFA 电机命令，不执行闭环测试，不让车轮在带负载状态下转动；
- 不把 `UART-DMA(pre-print)` 计数当作并发安全或实时性的证明；
- 不因单次日志异常而直接修改软件映射；必须先保存原始日志、复测并完成影响分析；
- 不在 Gate-0 未闭环前启动第二阶段的数据契约、命令队列、IMU 服务化或共享上下文拆分。

固件中的 `encdiag` 实现位于 `Application/app_debug.c` 的 `app_debug_hwmap_snapshot()`；菜单入口位于 `Application/Task/task_menu.c`。该诊断函数仅读取编码器和 UART 诊断状态，并明确声明“不输出 PWM、不发送电机命令、不清零编码器、不创建周期任务”。

## 4. 标准执行步骤

### 4.1 建立静态基线快照

1. 烧录已确认的基线固件并复位。
2. 保存完整启动日志，不只截取 `encdiag` 段落。
3. 不转动车轮，输入一次 `encdiag`。
4. 将输出保存为“静态基线快照”。首次快照的 `delta` 显示 `N/A` 属于设计行为。
5. 再次保持所有车轮静止，输入第二次 `encdiag`。
6. 记录第二次输出；四个逻辑轮的 `delta` 应为 0 或能由极少量机械抖动解释。无法解释的变化应按失败处理。

### 4.2 单轮正向映射验证

依次对 LF、LB、RF、RB 执行以下完全相同的操作；每完成一个轮位都先停止并保存日志。

1. 输入 `encdiag`，作为该轮的“转动前快照”。
2. 只用手缓慢转动目标物理轮约 1 至 3 圈；不得转动其他轮，不得施加电机驱动。
3. 停止转动，确认机械完全静止。
4. 输入 `encdiag`，作为该轮的“转动后快照”。
5. 从两个快照中记录四行 `total`、`delta`、`edge`、`last-dir`、`overflow`，并判断是否满足第 5 节的通过条件。

### 4.3 单轮反向一致性验证

仅当该轮的正向映射验证已经通过时，才执行反向验证：

1. 再输入一次 `encdiag` 作为反向前快照。
2. 只手动沿相反方向缓慢转动同一物理轮约 1 至 3 圈。
3. 停止后再次输入 `encdiag`。
4. 记录目标行 `delta` 与 `last-dir`；预期其变化方向与正向测试可区分。物理“正/反”命名由操作人定义，因此不强制其绝对正负号必须与 `cfg-sign` 同号。

### 4.4 UART 观测值的记录范围

每次快照都会打印 `UART-DMA(pre-print)`。该行只能作为“执行 `encdiag` 前 UART DMA 计数的现场观测”，需要记录但不得用于得出 UART 已安全、无竞态或无丢包的结论。

若出现 `busy` 计数增加、输出截断、乱码、终端卡死或快照迟迟不返回：

1. 停止继续发送命令；
2. 保存完整串口原始日志；
3. 记录触发时正在做的轮位和操作；
4. 将 Gate-0 标记为未通过，并回到第一阶段 UART P0 处理，不进行架构迁移。

## 5. 通过、失败与停止判定

### 5.1 单轮映射通过条件

对每一个物理轮，以下条件必须同时成立：

1. 该物理轮被手动转动后，**仅预期逻辑行**出现可重复、非零的 `delta` 或 `total` 变化；
2. 另外三行在同一观察窗口内的 `delta` 为 0，或有可记录、可重复且经人工排除的机械联动原因；
3. 目标行 `edge` 表明已观察到边沿，`last-dir` 在正反向手动转动中表现可区分；
4. `overflow` 不得因本次低速、短时手动转动而新增；
5. 静止复测不出现无原因累计值变化；
6. 串口输出完整，且不存在导致无法可靠判读的卡死、截断或乱码。

`cfg-sign` 是固件配置观察值，必须原样记录。除非项目另有明确的机械坐标系定义，本测试不以“手动正向必须显示正数”为通过条件；重点是**映射唯一性、反向可区分性与计数稳定性**。

### 5.2 立即失败 / 停止条件

出现任一情形即停止后续轮位测试，并将 Gate-0 标记为未通过：

- 一个物理轮转动导致两个或更多逻辑轮出现无法解释的非零变化；
- 预期逻辑轮始终不变化，或静止时持续变化；
- `overflow` 增加；
- 电机出现非预期动作；
- 串口诊断卡死、乱码、严重截断，或无法保存原始日志；
- 现场安全条件失效。

失败时只允许执行“保存证据、复位、静态影响分析、单独修复计划”四项动作；不得在同一工作包内尝试修改多个引脚、方向符号和控制参数来“碰运气”。

## 7. 验证完成后的受控回退

`encdiag` 是临时诊断，不应长期留在正式菜单与固件中。只有在下列条件全部满足后，才允许回退：

1. 第 6 节的原始证据已归档且由复核人确认；
2. 映射失败项（如有）已经通过独立提交修复并重新完成验证；
3. 第一阶段 P0 的其他准入证据已完成；
4. 后续工作包的负责人明确不再需要该临时命令。

回退动作必须是独立提交，使用：

```powershell
git revert --no-edit 55590bb
```

不得使用 `git reset`、覆盖性复制或手工删除历史。回退后必须进行一次 Keil 构建，并保存构建日志和新提交 SHA。若后续仍需临时观测能力，应另建新的只读诊断工作包，不能通过撤销回退或混入架构重构来恢复。
